# PRD｜沙城传话器 + 通知系统 + @AI 上下文对话

- 版本：v0.1
- 日期：2026-06-28
- 范围：沙城（候鸟300）主视图的通知系统重构、底部传话器、@AI 基于上下文（实时数据 / 知识库 RAG / 角色记忆）的对话能力

## 1. 背景与目标

原沙城主视图的「广播」「通知」分散在右侧手卷的两个 tab 内，玩家与世界之间只能"看"不能"说"。本期把界面改造成一个**能说话、能被回应、有知识有记忆**的世界界面：

- 玩家可在主视图直接向沙城传话（语音 / 文字），并可 @ 指定 AI 居民。
- 被 @ 的 AI 居民结合自身设定、沙城实时数据、候鸟300知识库与自身记忆作答。
- 广播与个人通知合并为统一的「通知系统」浮层，玩家的传话与 AI 回应都汇入其中。

## 2. 功能范围

1. 通知系统浮层（广播 + 通知合一）。
2. 底部传话器（语音 / 文字 / @ 角色）。
3. @AI → 异步上下文回应。
4. 候鸟300知识库 RAG（内置）。
5. 角色长期记忆接入。

不在本期：多人可见的公共对话广场、AI 主动发起对话、跨角色记忆共享、流式输出。

## 3. 详细设计

### 3.1 通知系统浮层 `src/components/BroadcastHud.tsx`

钉在主视图右上角的悬浮组件，取代原侧栏的「广播」「通知」两 tab。

- 数据合一：全城 AI 公开发言（`messages`）+ 个人通知（`notifications`）并入同一条按时间正序的流。
- 折叠态：单行（标题与最新一条合并显示）+ 未读红点（上限 `99+`）。
- 展开态：可滚动面板，自动滚到最新；顶部三筛选标签「全部 / 我的对话 / 广播」（我的对话 = 通知流，广播 = 全城发言）。
- 条目统一为单一 `FeedRow` 卡片模板，四类（广播 / 你的传话 / 角色回应 / 系统通知）仅靠头像色 + 小标签区分；未读用琥珀点 + 淡底；正文 `line-clamp-6`。
- 视觉：暖深色近不透明底，保证地图上可读。

### 3.2 传话器 `src/components/Composer.tsx`（样式见 `src/index.css` 传话器段）

主视图底部居中的"向沙城说话"器物，与世界美学同源（浮木炭色玻璃 + 陶土火漆发送印 + 苇笛语音 + 升起的纸笺选人）。

- 文字输入：自适应高度，Enter 发送 / Shift+Enter 换行。
- 语音输入：Web Speech API（zh-CN），实时转写；不支持的浏览器隐藏苇笛。
- @ 角色：输入 `@` 或点钮升起纸笺选择 AI 居民，作为本次传话对象。
- 提交：调用 `notifications.say`，落入通知系统时间流。
- 移动端：触控控制层（摇杆 / 动作键）激活时让位隐藏。

### 3.3 @AI 上下文回应 `convex/aiReply.ts`

`notifications.say` 写入 `user_said` 后，若带 `targetName` 则经 `scheduler.runAfter(0, …)` 异步触发 `replyAsCharacter`；非 AI 目标静默忽略。回应作为未读 `ai_reply` 通知回流，点亮红点。

发给 LLM 的上下文：

System Prompt（`gatherContext` 实时查询拼装）

| 段 | 内容 | 来源 |
| --- | --- | --- |
| 角色设定 | 身份 identity + 当下计划 plan | `agentDescriptions` |
| 环境氛围 | 最近 6 条全城广播 | `messages` |
| 沙城实时数据 | 作品数 / 分区 / AI 居民数 | `artworks` 实时 count、`world.agents` |
| 角色记忆 | 该角色 top-3 相关记忆 | `searchMemories` 向量检索 |
| 知识库 RAG | 候鸟300 top-4 文档块 | `kbChunks` 向量检索 |
| 行为指令 | 第一人称 1-3 句；数量用实时数据、背景用知识库、都没有就说不清楚 | 固定 |

对话历史：你与该角色的来往最近 8 轮（`notifications` 中 `user_said` / `ai_reply` 按 `userId × 角色名` 隔离），以 user/assistant 角色追加。

要点：以玩家最新一句做**一次 embedding**，同时驱动知识库 RAG 与角色记忆检索；模型走项目既有 `chatCompletion`（ZenMux 网关，env 配置），`max_tokens 220` / `temperature 0.9`。

### 3.4 候鸟300 知识库 RAG `convex/kb.ts` / `scripts/build-houniao-kb.mjs` / `convex/data/houniao300Kb.ts`

- 抽取：`build-houniao-kb.mjs` 读取 `houniao300-docs/content/docs` 的 23 篇 mdx，去 frontmatter / JSX / 图片，按标题+段落分块为 119 块，生成内置 TS 常量（运行时不依赖外部 repo）。
- 存储：`kbChunks` 向量表（1536 维向量索引）。
- 导入：`kb.ingest` 幂等（已有数据跳过，`force` 重建），批量嵌入入库。
- 检索：`retrieveKbByEmbedding(ctx, embedding, k)`，与角色记忆检索复用同一向量。

## 4. 数据模型（`convex/schema.ts`）

- `notifications`：`artworkId` / `artworkTitle` 改为可选；新增 `targetName`；`kind` 扩展 `user_said` / `ai_reply`。
- 新增 `kbChunks`：`chunkId`(唯一) / `source` / `title` / `text` / `embedding`，含 `chunkId` 普通索引 + `embedding` 向量索引。

新增后端函数：`notifications.say`、`aiReply.{gatherContext,insertReply,replyAsCharacter}`、`kb.{ingest,retrieveKb,retrieveKbByEmbedding,countChunks,clearAll,insertBatch,fetchChunks}`。

## 5. 验收（已自测通过）

- 数量类提问准确引用实时数据（「多少件艺术作品」→ 76 件，修复此前幻觉 23）。
- @AI 回应带入角色记忆与候鸟300知识（如黑客松"三百小时"来自知识库前言）。
- 你的传话与 AI 回应在通知系统正确显示、按 `userId × 角色` 隔离。
- 类型检查 `tsc` 与 `vite build` 通过。

## 6. 运维须知

- 每个部署需各跑一次 `npx convex run kb:ingest`（幂等，已有数据自动跳过）。
- 知识库更新：源文档改动后 `node scripts/build-houniao-kb.mjs` 重新生成内置块，再 `npx convex run kb:ingest '{"force":true}'` 重建向量。
- 共享 dev 部署（`exuberant-leopard-682`）：其它 worktree 的 stale codegen 可能覆盖 schema，注意。

## 7. 后续可选

- 检索相似度下限：低于阈值不注入知识库，避免"硬蹭资料"。
- 移动端传话器形态：收成小气泡按钮，与摇杆/动作键错位共存。
- 广播"有新发言"轻提示（不计入个人未读）。
- AI 回应流式输出 / 多角色被 @ 的群聊。
